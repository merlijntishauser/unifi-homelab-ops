# New Modules Design

Date: 2026-03-16

## Overview

Four new modules extending UniFi Homelab Ops beyond network analysis into physical infrastructure planning and documentation. Each strengthens the homelab niche by covering what the native UniFi dashboard does not provide.

| Module | Type | Controller dependency | Priority |
|--------|------|-----------------------|----------|
| Documentation Generator | Tightly connected | Live data | 1 |
| Rack Planner | Loosely connected | Optional enrichment | 2 |
| Cable & Port Mapper | Loosely connected | Pre-populated from LLDP | 3 |
| VLAN Planner | Tightly connected | Live data | 4 |

## Prerequisite: Library Refactoring

The Documentation Generator depends on rendering functions currently embedded in `unifi-network-maps`. These must move to `unifi-topology` so both the CLI tool and the web app consume the same library.

### What moves to `unifi-topology`

Generic rendering that produces Markdown or Mermaid output from topology model objects.

**Files** (from `unifi-network-maps/src/unifi_network_maps/render/`):

- `mermaid.py` -- Mermaid diagram generation
- `mermaid_theme.py` -- MermaidTheme dataclass and defaults
- `device_ports_md.py` -- per-device port overview as Markdown tables
- `device_ports_aggregate.py` -- port aggregation helpers
- `device_summary.py` -- port/PoE/uplink summary helpers
- `lldp_md.py` -- LLDP neighbor tables as Markdown
- `markdown_tables.py` -- generic Markdown table builder
- `legend.py` -- Mermaid legend rendering (generic part only)
- `templating.py` -- Jinja2 template loader
- Generic templates: `markdown_section.md.j2`, `device_port_block.md.j2`, `lldp_device_section.md.j2`, `mermaid_legend.mmd.j2`

**New public API on `unifi-topology`:**

```python
from unifi_topology.render import (
    # existing
    render_svg, render_svg_isometric, render_dual,
    # new
    render_mermaid,
    render_device_port_overview,
    render_lldp_tables,
    render_inventory_table,
    render_legend,
    MermaidTheme, DEFAULT_MERMAID_THEME,
)
```

All render functions accept model objects (`Device`, `Edge`, `TopologyResult`) and option dataclasses. No `argparse` dependency.

### What stays in `unifi-network-maps`

- CLI layer (argparse, main entry point)
- MkDocs-specific rendering (`mkdocs.py`, `mkdocs_document.md.j2`, dual theme styles, sidebar assets)
- Theme file loading (YAML theme files)
- IO/export (file writing, stdout)
- Mock data generation

### End-state architecture

```
unifi-topology (library)
  adapters/        API client, DNS, caching
  model/           Device, Edge, Firewall, VLANs, inventory
  render/
    svg.py                 SVG orthogonal
    svg_isometric.py       SVG isometric
    inventory.py           inventory table
    mermaid.py             Mermaid diagrams         (moved)
    markdown.py            Markdown tables/sections (moved)
    lldp.py                LLDP neighbor tables     (moved)
    templates/             Jinja2 templates         (moved)

unifi-network-maps (thin CLI shell)
  cli/             argparse, orchestration
  render/          MkDocs-specific assembly only
  themes/          YAML theme files

unifi-homelab-ops (web app)
  backend/         calls unifi_topology.render.* directly
  frontend/        renders output in web UI
```

## Module 5: Documentation Generator

### Purpose

A "living documentation" view that renders the entire network setup as a browsable, searchable document in the web UI. The same data the CLI generates as flat Markdown files, but interactive and always current.

### Sections

Each section is collapsible in the frontend.

1. **Network Map** -- Mermaid diagram rendered as SVG. Shows topology graph with device types, connections, port labels. Generated by `unifi_topology.render.render_mermaid()`, converted to SVG server-side or client-side.

2. **Infrastructure Inventory** -- sortable, searchable table of all devices: name, model, IP, MAC, firmware, uptime, uplink. Generated by `unifi_topology.render.render_inventory_table()`.

3. **Device Port Overview** -- per-device expandable sections with port tables: port number, speed, PoE, connected device, VLAN. Generated by `unifi_topology.render.render_device_port_overview()`.

4. **LLDP Neighbors** -- neighbor discovery tables for troubleshooting. Generated by `unifi_topology.render.render_lldp_tables()`.

5. **Firewall Summary** -- zone count, grade distribution, top findings. Pulled from existing firewall module data (homelab-ops native, not from `unifi-topology`).

6. **Metrics Snapshot** -- current device health: CPU/memory/temperature ranges, active notifications. Also homelab-ops native.

### Backend

New router: `/api/docs`

- `GET /api/docs/sections` -- returns JSON with pre-rendered Markdown strings per section
- `GET /api/docs/export` -- returns concatenated Markdown file for download

The backend calls `unifi-topology` render functions and supplements with homelab-ops-native data (firewall grades, metrics snapshots).

### Frontend

- New `DocumentationModule` component
- Renders Markdown to HTML using `react-markdown` (lightweight, already common in React ecosystems)
- Each section is a collapsible card with a header showing section name and item count
- Search bar filters across all sections (client-side text search)
- "Download Markdown" and "Print" buttons in the toolbar
- Auto-refreshes on the same polling cadence as other modules

### Data flow

```
Controller API
  --> unifi-topology (fetch + normalize)
    --> unifi-topology.render.* (Markdown strings)
      --> /api/docs/sections (JSON with sections)
        --> DocumentationModule (render Markdown as HTML)
```

## Module 6: Rack Planner

### Purpose

Visual drag-and-drop editor for designing 10" and 19" server racks. Place UniFi devices, patch panels, shelves, and third-party equipment. See power budget, heat output, and generate a bill of materials.

### Data model

Stored in SQLite. Fully local, no controller dependency.

```
Rack
  id              INTEGER PRIMARY KEY
  name            TEXT
  size            TEXT            -- "10-inch" | "19-inch"
  height_u        INTEGER         -- 6 to 48
  location        TEXT            -- optional label

RackItem
  id              INTEGER PRIMARY KEY
  rack_id         INTEGER REFERENCES Rack
  position_u      INTEGER         -- 1-based from bottom
  height_u        INTEGER         -- 1 to 4
  device_type     TEXT            -- gateway|switch|ap|patch-panel|shelf|ups|other
  label           TEXT
  power_watts     REAL            -- manual or auto-filled from device specs
  device_mac      TEXT            -- optional link to real device from topology
  notes           TEXT
```

### Views

**Rack Editor** (primary view)

- Front view of rack as a vertical CSS grid. Each row = 1U.
- Devices are draggable cards showing: label, model icon, U height, power draw.
- Color-coded by device type: gateway blue, switch teal, patch panel gray, shelf neutral.
- Empty slots show as subtle dashed outlines.
- Drag to reposition within the rack.
- Click to edit: label, power, notes, device link.

**Rack Overview**

- Grid of all racks (card per rack).
- Each card shows: rack name, fill percentage, total power, device count.
- Click to open the Rack Editor for that rack.
- "New Rack" button to create.

**Bill of Materials**

- Auto-generated list: what is in the rack, plus suggestions for missing items.
- Blanking plates for empty U slots.
- Suggested shelves for non-rackmount devices.
- Exportable as Markdown or CSV.

### Auto-populate

"Import from topology" action:

1. Fetches device list from existing topology module data.
2. Maps devices to rack items using model name to determine U height and device type.
3. Suggests placement: gateways near top, switches middle, patch panels bottom.
4. User confirms, adjusts positions, and saves.

### Power budget

- Sum `power_watts` per rack.
- For known UniFi models, auto-fill wattage from a built-in lookup table.
- User-configurable PDU limit per rack.
- Visual warning bar when approaching limit.

### Backend

New router: `/api/racks`

- CRUD operations for Rack and RackItem.
- `GET /api/racks` -- list all racks with summary stats.
- `GET /api/racks/:id` -- rack with all items, enriched with live device data if linked.
- `POST /api/racks/:id/import` -- auto-populate from topology.
- `GET /api/racks/:id/bom` -- bill of materials.

### Frontend

- Drag-and-drop with HTML5 Drag and Drop API (no additional library needed).
- Rack visualization is CSS grid, not canvas/SVG, keeping it accessible and printable.
- Responsive: on mobile, rack view scrolls vertically; editing is desktop-focused.

## Module 7: Cable & Port Mapper

### Purpose

Document the physical cable layer that UniFi cannot see. Which cable runs where, through which patch panels, in which colors and lengths. Bridges logical topology (UniFi knows "Switch Port 3 connects to AP-Office") and physical reality ("5m blue Cat6 through Patch Panel A, port 12").

### Data model

Stored in SQLite. Local data, pre-populated from controller LLDP edges.

```
Cable
  id              INTEGER PRIMARY KEY
  label           TEXT            -- e.g. "C-042"
  color           TEXT            -- blue|green|red|yellow|white|gray|orange|purple
  length_m        REAL
  cable_type      TEXT            -- Cat5e|Cat6|Cat6a|fiber-sm|fiber-mm|DAC
  notes           TEXT

CableRun
  id              INTEGER PRIMARY KEY
  cable_id        INTEGER REFERENCES Cable
  from_device     TEXT            -- MAC address or rack_item_id
  from_port       INTEGER
  to_device       TEXT
  to_port         INTEGER
  patch_panel_id  INTEGER REFERENCES PatchPanel (nullable)
  patch_port      INTEGER         -- port on the patch panel (nullable)

PatchPanel
  id              INTEGER PRIMARY KEY
  rack_item_id    INTEGER REFERENCES RackItem (nullable, links to Rack Planner)
  name            TEXT
  port_count      INTEGER
  panel_type      TEXT            -- keystone|fiber|rj45
```

### Views

**Port Map Table** (primary view)

- Per-device expandable table.
- Each port row: port number, speed (live from controller), connected device (live), cable label, cable color (visual dot), patch panel hop, notes.
- Pre-populated from topology LLDP edges. User fills in physical details.

**Patch Panel View**

- Visual front view of a patch panel: row of numbered ports.
- Color-coded by cable color.
- Click a port to see or edit the cable run.
- Same visual language as rack planner (can be embedded in rack editor view).

**Cable List**

- Filterable table of all cables: label, type, length, color, from, to.
- Useful for ordering spares or auditing.

### Enrichment from topology

Pre-populates `CableRun.from_device`, `from_port`, `to_device`, `to_port` from the controller's LLDP edge data. User adds the physical details (cable color, length, patch panel hops).

### Cross-module links

- **Rack Planner**: patch panels placed in racks link to their port views here.
- **Topology Module**: device panel gains a "Physical connections" tab pulling from cable data.
- **Documentation Generator**: cable inventory becomes a section in generated docs.

### Backend

New router: `/api/cables`

- CRUD for Cable, CableRun, PatchPanel.
- `GET /api/cables/by-device/:mac` -- cable runs for a specific device.
- `POST /api/cables/import` -- pre-populate from topology edges.
- `GET /api/cables/export` -- Markdown or CSV export.

### Frontend

- Port Map Table is the default view (most practical for daily use).
- Patch Panel View reuses the CSS grid pattern from Rack Planner.
- Cable color shown as colored dots/pills (same visual pattern as status indicators).

## Module 8: VLAN Planner

### Purpose

Visual subnet map showing VLAN allocations, IP utilization, zone assignments, and conflict detection. UniFi shows VLANs as a flat list -- this shows relationships and helps plan changes before applying them.

### Data sources

All live from controller via `unifi-topology`:

- Networks (VLAN ID, subnet, DHCP range, purpose)
- Firewall zones (which networks belong to which zone)
- Client list (which IPs are in use, on which VLAN)
- Device list (management IPs, per-port VLAN assignments)

### Views

**Subnet Map** (primary view)

- Visual block diagram.
- Each VLAN is a horizontal bar scaled by subnet size (/24 narrow, /16 wide).
- Color-coded by firewall zone.
- Shows: VLAN ID, name, subnet, IP utilization fill bar (used/total).
- Sorted by VLAN ID or grouped by zone (toggle).

**IP Address Table**

- Per-VLAN expandable view.
- Every IP in use: address, hostname (DNS resolved via `unifi-topology`), MAC, device type, last seen.
- Sortable, searchable.

**Conflict Detector**

Automated checks surfaced as findings (same severity pattern as firewall analysis):

- Overlapping subnets
- VLANs not assigned to any firewall zone (orphaned)
- Zones with no networks
- Oversized subnets for homelab use (/16 when /24 would suffice)
- DHCP range conflicts
- Duplicate static IP assignments

**Zone-VLAN Matrix**

- Small cross-reference grid: zones as rows, VLANs as columns.
- Clicking a zone navigates to the firewall module for that zone's pairs.

### Read-only

This module only reads from the controller. No write operations. Users plan changes here and apply them in the UniFi dashboard.

### Relationship to existing modules

- **Firewall Module**: zone names and zone-pair grades feed into the Zone-VLAN Matrix.
- **Documentation Module**: VLAN table and subnet map become sections in generated docs.
- **Topology Module**: device port VLAN assignments enrich the subnet map.

### Backend

New router: `/api/vlans`

- `GET /api/vlans/subnets` -- subnet map data with utilization.
- `GET /api/vlans/:id/ips` -- IP address table for a specific VLAN.
- `GET /api/vlans/conflicts` -- conflict detection results.
- `GET /api/vlans/zone-matrix` -- zone-to-VLAN cross-reference.

Data is fetched via `unifi-topology` (`fetch_networks`, `fetch_clients`, `fetch_firewall_zones`) and enriched with DNS resolution (`resolve_hostnames`).

## Implementation sequence

### Phase 1: Library refactoring (unifi-topology + unifi-network-maps)

1. Move generic rendering from `unifi-network-maps` to `unifi-topology`.
2. Update `unifi-network-maps` to import from `unifi-topology.render`.
3. Release new versions of both libraries.
4. Update `unifi-homelab-ops` dependency.

### Phase 2: Documentation Generator

5. Add `/api/docs` router calling `unifi-topology.render.*`.
6. Build `DocumentationModule` frontend with `react-markdown`.
7. Add firewall summary and metrics snapshot sections.
8. Add export (Markdown download).

### Phase 3: Rack Planner

9. Database schema (Rack, RackItem).
10. `/api/racks` CRUD router.
11. Rack Editor frontend (CSS grid + drag-and-drop).
12. Auto-populate from topology.
13. Bill of materials view.

### Phase 4: Cable & Port Mapper

14. Database schema (Cable, CableRun, PatchPanel).
15. `/api/cables` CRUD router.
16. Port Map Table frontend.
17. Patch Panel View frontend.
18. Pre-populate from LLDP edges.
19. Cross-link with Rack Planner.

### Phase 5: VLAN Planner

20. `/api/vlans` router with subnet data, IP tables, conflict detection.
21. Subnet Map frontend.
22. IP Address Table frontend.
23. Conflict Detector.
24. Zone-VLAN Matrix.

Each phase is independently shippable. Phases 3 and 4 could be parallelized since they share no backend dependencies (only the optional cross-links added afterward).
